# بنية أوراكل: كيف تعمل قاعدة بيانات أوراكل

> يشرح هذا المستند البنية الداخلية لقاعدة بيانات أوراكل. فهم هذه المفاهيم أمر أساسي لكل مدير قاعدة بيانات (DBA)؛ فبدون هذه المعرفة، ستكون الأوامر التي تنفذها مجرد "سحر أسود" لا تفهم كنهه.

---

## 1. نظرة عامة

تتكون قاعدة بيانات أوراكل من جزأين أساسيين:

```
┌─────────────────────────────────────────────────────────────────┐
│                     ORACLE INSTANCE (النسخة)                   │
│          (هياكل في الذاكرة RAM + عمليات خلفية Background)        │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                   SGA (System Global Area)               │    │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────────┐  │    │
│  │  │  Shared  │ │ Database │ │   Redo   │ │   Large   │  │    │
│  │  │  Pool    │ │ Buffer   │ │   Log    │ │   Pool    │  │    │
│  │  │          │ │ Cache    │ │  Buffer  │ │           │  │    │
│  │  └──────────┘ └──────────┘ └──────────┘ └───────────┘  │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ┌──────────┐  (لكل جلسة مستخدم منطقة PGA خاصة بها)             │
│  │ PGA Pool │                                                   │
│  └──────────┘                                                   │
│                                                                 │
│ Processes: PMON, SMON, DBWn, LGWR, CKPT, ARCn, MMON, RECO...    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │  قراءة / كتابة
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                     DATABASE (الملفات على القرص)                 │
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │  Datafile     │  │  Online Redo │  │  Controlfile │          │
│  │  (.dbf)       │  │  Log (.log)  │  │  (.ctl)      │          │
│  │               │  │              │  │              │          │
│  │  SYSTEM,      │  │  Group 1     │  │  بيانات      │          │
│  │  SYSAUX,      │  │  Group 2     │  │  وصفية       │          │
│  │  USERS,       │  │  Group 3     │  │              │          │
```

> **مفهوم جوهري**: الـ **INSTANCE** موجودة في الذاكرة (متطايرة). الـ **DATABASE** موجودة على القرص (دائمة). عندما تقول "قاعدة البيانات متوقفة"، فأنت تقصد أن الـ instance قد توقفت. أما الملفات على القرص فهي موجودة دائماً.

---

## 2. منطقة الـ SGA (System Global Area) — قلب الذاكرة

### 2.1 Database Buffer Cache (مخزن البيانات المؤقت)
- **ما هو؟** ذاكرة RAM تخزن كتل البيانات (Data Blocks) المقروءة من القرص.
- **كيف يعمل؟** عندما يطلب استعلام قراءة جدول، لا يذهب أوراكل للقرص مباشرة. أولاً، يبحث في الـ Buffer Cache. إذا وجدها (**Cache Hit**) تكون السرعة فائقة. إذا لم يجدها (**Cache Miss**) يضطر للقراءة من القرص.
- **كتل ملوثة (Dirty) مقابل نظيفة (Clean)**: الكتلة "الملوثة" هي التي تم تعديلها في الذاكرة ولم تكتب بعد على القرص.

### 2.2 Shared Pool
- تحتوي على الـ **Library Cache** (الاستعلامات التي تم تحليلها وخطط التنفيذ) والـ **Data Dictionary Cache** (بيانات الجداول والأعمدة والمستخدمين). هذا يمنع تكرار تحليل نفس الاستعلام عدة مرات.

### 2.3 Redo Log Buffer
- ذاكرة مؤقتة دائرية تسجل فيها كافة التغييرات التي تتم على البيانات قبل كتابتها في ملفات الـ Redo Log على القرص لضمان القدرة على استعادة البيانات في حال تعطل النظام.

---

## 3. منطقة الـ PGA (Program Global Area) — الذاكرة الخاصة
- لكل جلسة مستخدم (اتصال) منطقة PGA خاصة بها غير مشتركة.
- تحتوي على مناطق الفرز (Sorting) والـ Hash Join ومتغيرات الجلسة.

---

## 4. العمليات الخلفية (Background Processes) — "العمال" غير المرئيين

- **DBWn (Database Writer)**: يكتب الكتل المعدلة (Dirty) من الذاكرة إلى ملفات البيانات على القرص.
- **LGWR (Log Writer)**: يكتب سجلات التغيير من الذاكرة إلى ملفات الـ Redo Log على القرص (عند الـ COMMIT).
- **CKPT (Checkpoint)**: يضمن مزامنة البيانات وتحديث رؤوس الملفات.
- **SMON/PMON**: مسؤولان عن مراقبة النظام والعمليات وتنظيف الموارد بعد الأعطال.

---

## 5. ملفات قاعدة البيانات — الفصل بين المنطقي والفيزيائي

- **الهيكل المنطقي**: يبدأ من الـ **Tablespace** (مثل USERS, SYSTEM) الذي يحتوي على **Segments** (جداول، فهارس)، والتي تتكون من **Extents**، وهذه الأخيرة هي مجموعة من الـ **Blocks**.
- **الهيكل الفيزيائي**: هي الملفات الحقيقية على القرص (.dbf, .log, .ctl).

---

## 6. سجلات الإعادة (Redo Log) — "يوميات" التغييرات
- تعمل بشكل دائري. عندما يمتلئ ملف، ينتقل النظام للكتابة في الملف التالي (**Log Switch**).
- في وضع **ARCHIVELOG**، يتم نسخ الملف الممتلئ إلى مكان آمن قبل إعادة الكتابة عليه، وهو أمر حيوي لعمل Data Guard وعمليات الاستعادة.

---

## 7. مقاطع التراجع (Undo Segments) — "السفر عبر الزمن"
- الـ **Undo** هو المكان الذي يحفظ فيه أوراكل النسخة "القديمة" من البيانات قبل تعديلها.
- **فوائده**: 
  1. التراجع عن العمليات (**Rollback**).
  2. تناسق القراءة (**Read Consistency**): المستخدم (ب) يرى البيانات القديمة طالما أن المستخدم (أ) لم يقم بعمل COMMIT بعد.
  3. استعادة النظام بعد الانهيار.

---

## 8. نظام ASM (Automatic Storage Management)
- هو نظام أوراكل الذكي لإدارة الأقراص. يقوم بتوزيع البيانات على الأقراص بشكل متوازي (**Striping**) وتوفير نسخ احتياطية لها (**Mirroring**).

---

## 9. بنية RAC (Real Application Clusters)
- تسمح لعدة نسخ (Instances) بالعمل على نفس قاعدة البيانات (على قرص مشترك).
- تستخدم تقنية **Cache Fusion** لمشاركة البيانات في الذاكرة عبر شبكة الربط الداخلي (Interconnect) لزيادة الأداء والتوافر العالي.
