# Guida Studio Colloquio Oracle DBA

Questa guida accompagna lo schema:

- [.schema_oracle_dba_lab_colloquio.sql](./.schema_oracle_dba_lab_colloquio.sql)

Serve per sapere cosa studiare, cosa testare nello schema e quali runbook aprire.

## Obiettivo Del Lab

Lo schema `DBA_LAB` simula una piccola applicazione enterprise con:

- clienti;
- prodotti;
- ordini;
- righe ordine;
- pagamenti;
- audit log;
- batch/load run.

Le tabelle principali sono:

```text
LAB_CUSTOMERS
LAB_PRODUCTS
LAB_ORDERS
LAB_ORDER_ITEMS
LAB_PAYMENTS
LAB_AUDIT_LOG
LAB_BATCH_EVENTS
LAB_LOAD_RUNS
```

Alcuni indici sono volutamente mancanti per creare casi di tuning:

```text
LAB_ORDERS(CUSTOMER_ID)
LAB_ORDER_ITEMS(ORDER_ID)
```

Query da testare:

```sql
SELECT c.region, p.category, COUNT(*) orders_count, SUM(oi.line_amount) total_amount
FROM lab_orders o
JOIN lab_customers c ON c.customer_id = o.customer_id
JOIN lab_order_items oi ON oi.order_id = o.order_id
JOIN lab_products p ON p.product_id = oi.product_id
WHERE o.order_date >= TRUNC(SYSDATE) - 180
  AND o.status IN ('PAID','SHIPPED')
GROUP BY c.region, p.category
ORDER BY total_amount DESC;
```

Dopo averla eseguita:

```sql
SELECT *
FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'ALLSTATS LAST +OUTLINE +ALIAS'));
```

Cosa devi spiegare:

```text
Guardo il piano reale, non solo il costo. Controllo join order, access path,
cardinalita stimate e reali, buffer gets, letture fisiche e uso TEMP. Se vedo
full scan o nested loop inefficiente, verifico statistiche e indici mancanti.
```

Fix da provare solo dopo il before:

```sql
CREATE INDEX ix_lab_orders_customer_id ON lab_orders(customer_id);
CREATE INDEX ix_lab_items_order_id ON lab_order_items(order_id);

BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(USER, cascade => TRUE);
END;
/
```

Validazione after:

```sql
SELECT *
FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'ALLSTATS LAST +OUTLINE +ALIAS'));
```

Risposta colloquio:

```text
Non creo un indice alla cieca. Misuro before/after: piano, elapsed time,
buffer gets, physical reads, cardinalita e impatto DML. Se il fix e urgente
preferisco soluzioni reversibili come SQL Plan Baseline o SQL Patch.
```

## 3. Predicate Non Sargable

Query problematica:

```sql
SELECT COUNT(*), SUM(total_amount)
FROM lab_orders
WHERE TRUNC(order_date) = TRUNC(SYSDATE) - 30;
```

Variante migliore:

```sql
SELECT COUNT(*), SUM(total_amount)
FROM lab_orders
WHERE order_date >= TRUNC(SYSDATE) - 30
  AND order_date <  TRUNC(SYSDATE) - 29;
```

Cosa devi spiegare:

```text
Applicare una funzione alla colonna puo impedire l'uso efficiente dell'indice.
Riscrivo il predicato come range sulla colonna nativa. In alternativa potrei
valutare un function-based index, ma solo se il pattern e stabile e giustificato.
```

## 6.1 Backup RMAN Live Da Forzare

Obiettivo:

```text
Lanciare un backup manuale e vedere in tempo reale:
- cosa e' stato backuppato;
- se controlfile e SPFILE sono inclusi;
- se gli archivelog sono presenti;
- se il restore e' validabile;
- se la FRA ha spazio o file reclaimable.
```

Da shell:

```bash
rman target /
```

Fotografia prima del backup:

```sql
SHOW ALL;
LIST BACKUP SUMMARY;
REPORT SCHEMA;
REPORT NEED BACKUP;
```

Come lo spieghi:

```text
Prima di lanciare un backup controllo configurazione, retention, device type,
controlfile autobackup, stato dei backup esistenti e file che non rispettano la
policy. Cosi capisco cosa c'e' e cosa manca.
```

Backup manuale database + archivelog + controlfile + SPFILE:

```sql
RUN {
  SQL "ALTER SYSTEM ARCHIVE LOG CURRENT";
  BACKUP AS COMPRESSED BACKUPSET DATABASE TAG 'DBA_LAB_MANUAL_DB';
  BACKUP AS COMPRESSED BACKUPSET ARCHIVELOG ALL TAG 'DBA_LAB_MANUAL_ARCH';
  BACKUP CURRENT CONTROLFILE TAG 'DBA_LAB_MANUAL_CTL';
  BACKUP SPFILE TAG 'DBA_LAB_MANUAL_SPFILE';
}
```

Controlli immediati in RMAN:

```sql
LIST BACKUP TAG 'DBA_LAB_MANUAL_DB';
LIST BACKUP TAG 'DBA_LAB_MANUAL_ARCH';
LIST BACKUP OF CONTROLFILE;
LIST BACKUP OF SPFILE;
LIST ARCHIVELOG ALL;
RESTORE DATABASE PREVIEW SUMMARY;
RESTORE DATABASE VALIDATE;
RESTORE ARCHIVELOG ALL VALIDATE;
```

Come lo spieghi:

```text
Dopo il backup non mi fermo allo status completed. Controllo i backup pieces,
gli archivelog, il controlfile, lo SPFILE e faccio restore validate. In un
colloquio enterprise questo dimostra che penso alla recovery, non solo al backup.
```

Controlli da SQL*Plus:

```sql
SELECT input_type,
       status,
       TO_CHAR(start_time, 'YYYY-MM-DD HH24:MI:SS') AS start_time,
       TO_CHAR(end_time, 'YYYY-MM-DD HH24:MI:SS') AS end_time,
       output_bytes_display
FROM v$rman_backup_job_details
WHERE start_time > SYSDATE - 1
ORDER BY start_time DESC;
```

```sql
SELECT file_type,
       percent_space_used,
       percent_space_reclaimable,
       number_of_files
FROM v$flash_recovery_area_usage
ORDER BY percent_space_used DESC;
```

```sql
SELECT name,
       ROUND(space_limit/1024/1024/1024, 2) AS limit_gb,
       ROUND(space_used/1024/1024/1024, 2) AS used_gb,
       ROUND(space_reclaimable/1024/1024/1024, 2) AS reclaimable_gb
FROM v$recovery_file_dest;
```

Se vuoi vedere backup falliti o warning:

```sql
SELECT input_type,
       status,
       TO_CHAR(start_time, 'YYYY-MM-DD HH24:MI:SS') AS start_time,
       TO_CHAR(end_time, 'YYYY-MM-DD HH24:MI:SS') AS end_time,
       error_count
FROM v$rman_backup_job_details
WHERE start_time > SYSDATE - 7
ORDER BY start_time DESC;
```

Se devi verificare archivelog generati di recente:

```sql
SELECT thread#,
       sequence#,
       first_time,
       next_time,
       applied,
       deleted,
       status
FROM v$archived_log
WHERE first_time > SYSDATE - 1
ORDER BY thread#, sequence#;
```

Nota importante:

```text
Non usare DELETE ARCHIVELOG o DELETE OBSOLETE in un ambiente reale senza sapere
retention, Data Guard, GoldenGate e policy aziendale. In colloquio dillo
esplicitamente: pulire spazio non deve distruggere la recovery chain.
```

## 6.2 Restore Point E Flashback

Obiettivo:

```text
Sapere quando usare Flashback invece di un restore RMAN completo.
Flashback e' utile per rollback veloci dopo change, deploy, patch, test o errore
logico recente, se la configurazione e la retention lo permettono.
```

### Verifiche iniziali

Da SQL*Plus come SYSDBA:

```sql
SELECT name, open_mode, log_mode, flashback_on
FROM v$database;
```

```sql
SHOW PARAMETER db_recovery_file_dest
SHOW PARAMETER db_recovery_file_dest_size
SHOW PARAMETER db_flashback_retention_target
```

```sql
SELECT name,
       ROUND(space_limit/1024/1024/1024, 2) AS limit_gb,
       ROUND(space_used/1024/1024/1024, 2) AS used_gb,
       ROUND(space_reclaimable/1024/1024/1024, 2) AS reclaimable_gb
FROM v$recovery_file_dest;
```

Come lo spieghi:

```text
Prima verifico che il database sia in ARCHIVELOG, che Flashback sia attivo o
attivabile, che la FRA abbia spazio e che la retention sia coerente con la
finestra temporale richiesta.
```

### Attivare Flashback Database

Solo in lab o change approvato:

```sql
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE ARCHIVELOG;
ALTER DATABASE FLASHBACK ON;
ALTER DATABASE OPEN;
```

Verifica:

```sql
SELECT flashback_on
FROM v$database;
```

### Creare un guaranteed restore point

```sql
CREATE RESTORE POINT rp_before_dba_lab_test GUARANTEE FLASHBACK DATABASE;
```

Verifica:

```sql
SELECT name,
       scn,
       time,
       guarantee_flashback_database,
       storage_size
FROM v$restore_point
ORDER BY time DESC;
```

Come lo spieghi:

```text
Un guaranteed restore point forza Oracle a mantenere i flashback logs necessari
per tornare a quel punto. E' molto utile prima di patch o test invasivi, ma va
monitorato perche puo consumare FRA.
```

### Test controllato con lo schema DBA_LAB

Prima del danno:

```sql
SELECT COUNT(*) AS orders_before
FROM lab_orders;
```

Simula errore logico:

```sql
DELETE FROM lab_orders
WHERE order_date < TRUNC(SYSDATE) - 700;

COMMIT;
```

Verifica impatto:

```sql
SELECT COUNT(*) AS orders_after_delete
FROM lab_orders;
```

### Flashback Query

Se vuoi solo leggere il dato com'era prima:

```sql
SELECT COUNT(*) AS orders_10_min_ago
FROM lab_orders AS OF TIMESTAMP (SYSTIMESTAMP - INTERVAL '10' MINUTE);
```

Ripristino selettivo con insert da flashback query:

```sql
INSERT INTO lab_orders
SELECT *
FROM lab_orders AS OF TIMESTAMP (SYSTIMESTAMP - INTERVAL '10' MINUTE) old_o
WHERE NOT EXISTS (
  SELECT 1
  FROM lab_orders cur_o
  WHERE cur_o.order_id = old_o.order_id
);

COMMIT;
```

Come lo spieghi:

```text
Flashback Query e' non invasivo: leggo una versione precedente dei dati e posso
ricostruire solo le righe mancanti, se undo retention e volume lo consentono.
```

### Flashback Table

Prerequisito: row movement abilitato sulla tabella.

```sql
ALTER TABLE lab_orders ENABLE ROW MOVEMENT;
ALTER TABLE lab_order_items ENABLE ROW MOVEMENT;
ALTER TABLE lab_payments ENABLE ROW MOVEMENT;
```

Esempio:

```sql
FLASHBACK TABLE lab_orders TO TIMESTAMP (SYSTIMESTAMP - INTERVAL '10' MINUTE);
```

Nota:

```text
Con vincoli e tabelle correlate devo fare attenzione. Se flashbacko solo la
tabella padre o solo la figlia posso creare incoerenze logiche. In produzione
valuto dipendenze, foreign key, finestre temporali e test su clone.
```

### Flashback Database Al Restore Point

Procedura invasiva: riporta l'intero database al restore point.

```sql
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
FLASHBACK DATABASE TO RESTORE POINT rp_before_dba_lab_test;
ALTER DATABASE OPEN RESETLOGS;
```

Validazione:

```sql
SELECT COUNT(*) AS orders_after_flashback
FROM lab_orders;

SELECT name, open_mode, database_role, flashback_on
FROM v$database;
```

Come lo spieghi:

```text
Flashback Database e' potente ma impatta tutto il database e richiede OPEN
RESETLOGS. Lo uso per rollback di change controllati o incidenti estesi, non per
una singola riga se posso usare Flashback Query/Table o RMAN RECOVER TABLE.
```

### Pulizia restore point

Dopo il test:

```sql
DROP RESTORE POINT rp_before_dba_lab_test;
```

Verifica FRA:

```sql
SELECT file_type,
       percent_space_used,
       percent_space_reclaimable,
       number_of_files
FROM v$flash_recovery_area_usage
ORDER BY percent_space_used DESC;
```

Risposta colloquio:

```text
Prima di un change critico creo un guaranteed restore point se lo spazio FRA lo
consente. Dopo il change valido applicazione e database. Se serve rollback posso
fare Flashback Database. Per errori logici piccoli preferisco Flashback Query o
Flashback Table, perche sono piu granulari.
```

## 9. Risposte Brevi Da Memorizzare

### SQL tuning

```text
SQL tuning non e creare subito un indice. E identificare SQL_ID, piano reale,
wait event, cardinalita, storico AWR e scegliere un fix misurabile.
```

### RMAN

```text
Un backup vale solo se ho prova di restore: validate, archivelog, retention,
controlfile autobackup e RTO/RPO chiari.
```

### Data Guard

```text
Data Guard riduce RTO, RMAN protegge la recovery storica. Sono complementari.
```

### Produzione

```text
In produzione salvo evidenze, applico fix reversibili quando possibile e chiudo
con validazione before/after e prevenzione.
```
